# Test Strategy

이 문서는 커피 주문 및 포인트 결제 시스템의 테스트 범위와 검증 기준을 정의한다.

비즈니스 규칙은 `BUSINESS_RULES.md`,  
API 형식은 `API_SPEC.md`,  
시스템 구조와 트랜잭션 전략은 `ARCHITECTURE.md`,  
코드 작성 규칙은 `CODE_CONVENTION.md`를 기준으로 한다.

본 프로젝트의 테스트는 단순 API 성공 여부뿐 아니라  
주문, 포인트, 멱등성, 동시성 및 외부 이벤트 전송의 정합성을 검증하는 것을 목표로 한다.

---

# 1. 테스트 목표

본 프로젝트에서는 다음 내용을 우선 검증한다.

- 회원, 메뉴, 장바구니, 포인트 및 주문 기능이 정상 동작하는가?
- 잘못된 요청과 비즈니스 규칙 위반을 올바르게 차단하는가?
- 포인트 차감과 주문 저장이 하나의 트랜잭션으로 처리되는가?
- 같은 회원의 동시 주문에서 포인트가 초과 차감되지 않는가?
- 주문 중 장바구니와 메뉴 정보가 변경되지 않는가?
- 동일한 멱등키 요청이 여러 번 들어와도 주문이 한 번만 생성되는가?
- 외부 Mock API 장애가 완료된 주문을 Rollback하지 않는가?
- Outbox와 Kafka 구조에서 이벤트가 유실되거나 중복 처리되지 않는가?
- 동시성 전략과 동기·비동기 구조의 성능 차이를 측정할 수 있는가?

---

# 2. 테스트 종류

테스트는 다음 단계로 구분한다.

```text
단위 테스트
→ Repository 테스트
→ 통합 테스트
→ 동시성 테스트
→ 외부 연동 테스트
→ 성능 테스트
```

| 테스트 종류 | 목적 |
|---|---|
| 단위 테스트 | Entity와 도메인 규칙 검증 |
| Repository 테스트 | JPA Query와 잠금 조회 검증 |
| 통합 테스트 | Controller부터 DB까지 실제 흐름 검증 |
| 동시성 테스트 | 여러 요청이 동시에 실행될 때 정합성 검증 |
| 외부 연동 테스트 | Mock API 지연, 실패 및 Timeout 검증 |
| 이벤트 테스트 | Outbox, Kafka 발행 및 Consumer 처리 검증 |
| 성능 테스트 | 응답시간, 처리량 및 락 전략 비교 |

# 3. 테스트 환경

## 3.1 기본 환경

로컬 테스트 환경은 Docker Compose를 기준으로 구성한다.

```text
Spring Boot
MySQL
Redis
Mock API
Kafka
```

1차 구현에서는 다음 서비스를 사용한다.

```text
Spring Boot
MySQL
Redis
Mock API
```

2차 고도화에서는 다음 서비스를 추가한다.

```text
Kafka
Kafka Consumer
Outbox Publisher
```

---

## 3.2 테스트 DB

단순 Entity 테스트를 제외한 Repository, 트랜잭션 및 동시성 테스트는  
실제 운영 환경과 동일한 MySQL을 사용한다.

H2는 다음 항목에서 MySQL과 동작 차이가 발생할 수 있으므로  
핵심 정합성 테스트에는 사용하지 않는다.

```text
비관적 락
트랜잭션 격리 수준
Unique 제약조건
Deadlock
동시 UPDATE
QueryDSL SQL
```

권장 방식:

```text
Testcontainers MySQL
또는
Docker Compose MySQL Test DB
```

---

## 3.3 Redis 테스트

다음 기능은 실제 Redis 환경에서 검증한다.

- Access Token Blacklist
- Refresh Token Whitelist
- 로그아웃 후 토큰 차단
- Refresh Token 삭제
- TTL 적용

Redis Repository나 Serializer만 확인하는 단순 테스트는  
Testcontainers Redis 또는 별도 테스트 Redis를 사용할 수 있다.

---

## 3.4 Kafka 테스트

Kafka 테스트는 2차 고도화 시 진행한다.

권장 환경:

```text
Testcontainers Kafka
또는
Docker Compose Kafka
```

Kafka 테스트에서는 단순 메시지 발행 성공 여부뿐 아니라  
Consumer 처리 결과와 최종 DB 또는 외부 Mock API 호출 여부까지 검증한다.

# 4. 테스트 작성 규칙

테스트 클래스 이름은 다음 기준을 사용한다.

```text
{Target}Test
{Target}RepositoryTest
{Target}IntegrationTest
{Target}ConcurrencyTest
{Target}PerformanceTest
```

예시:

```text
PointTest
MenuRepositoryTest
OrderIntegrationTest
OrderConcurrencyTest
OrderPerformanceTest
```

테스트 메서드는 한글로 작성한다.

```java
@Test
void 포인트가_부족하면_주문에_실패한다() {
}
```

테스트는 가능한 한 Given-When-Then 구조를 따른다.

```java
@Test
void 판매중인_메뉴를_장바구니에_추가할_수_있다() {
    // given

    // when

    // then
}
```

# 5. 단위 테스트

단위 테스트는 DB나 외부 시스템 없이  
Entity와 도메인 규칙을 빠르게 검증한다.

## 5.1 Member

검증 항목:

- 회원 생성 시 기본 상태가 정상적으로 설정되는가?
- 이미 삭제된 회원을 다시 삭제할 수 없는가?
- 탈퇴 회원이 로그인 가능한 상태로 남지 않는가?
- 권한이 정상적으로 설정되는가?

예시:

```text
회원 생성 시 상태는 ACTIVE이다.
회원 삭제 시 deletedAt이 설정된다.
삭제 회원은 비활성 상태가 된다.
```

---

## 5.2 Menu

검증 항목:

- 메뉴 생성 시 기본 상태가 판매중인가?
- 가격이 0원 이하이면 생성할 수 없는가?
- 메뉴 이름이 비어 있으면 생성할 수 없는가?
- 메뉴 가격을 변경할 수 있는가?
- 메뉴 상태를 변경할 수 있는가?
- 삭제된 메뉴는 판매 상태로 변경할 수 없는가?

예시:

```text
정상 가격으로 메뉴를 생성할 수 있다.
음수 가격으로 메뉴를 생성하면 실패한다.
판매중 메뉴를 품절 상태로 변경할 수 있다.
삭제된 메뉴는 다시 판매할 수 없다.
```

---

## 5.3 Cart

검증 항목:

- 장바구니에 메뉴를 추가할 수 있는가?
- 동일 메뉴를 추가하면 수량이 증가하는가?
- 수량을 1 이상으로만 변경할 수 있는가?
- 장바구니 항목을 삭제할 수 있는가?
- 빈 장바구니로 주문할 수 없는가?

예시:

```text
동일 메뉴를 두 번 추가하면 하나의 항목으로 합쳐진다.
수량을 0으로 변경하면 실패한다.
빈 장바구니 주문은 실패한다.
```

---

## 5.4 Point

검증 항목:

- 포인트를 정상 충전할 수 있는가?
- 0 이하 금액은 충전할 수 없는가?
- 보유 포인트 내에서 사용할 수 있는가?
- 보유 포인트보다 많은 금액을 사용할 수 없는가?
- 포인트 사용 후 잔액이 정확한가?
- 잔액이 음수가 되지 않는가?

예시:

```text
10,000P 충전 후 잔액은 10,000P이다.
10,000P에서 3,000P 사용 후 잔액은 7,000P이다.
10,000P에서 11,000P 사용은 실패한다.
```

---

## 5.5 Order

검증 항목:

- 주문 항목 금액 합계가 주문 총금액과 일치하는가?
- 주문 항목이 없는 주문을 생성할 수 없는가?
- 주문 시점의 메뉴 이름과 가격을 저장하는가?
- 주문 상태가 정상적으로 설정되는가?
- 결제 금액과 주문 총금액이 일치하는가?

예시:

```text
메뉴 가격과 수량을 기준으로 총금액을 계산한다.
메뉴 가격이 변경되어도 기존 주문 항목 가격은 변경되지 않는다.
주문 총금액과 결제 금액이 다르면 실패한다.
```

# 6. Repository 테스트

Repository 테스트는 실제 MySQL을 사용해  
Query, Unique 제약조건 및 잠금 조회를 검증한다.

## 6.1 MemberRepository

검증 항목:

- 이메일로 회원을 조회할 수 있는가?
- 동일 이메일 회원이 중복 저장되지 않는가?
- 삭제되지 않은 회원만 조회되는가?

---

## 6.2 MenuRepository

검증 항목:

- 판매중인 메뉴 목록만 조회되는가?
- 삭제된 메뉴가 일반 조회에서 제외되는가?
- 카테고리와 상태 조건 조회가 정상인가?
- 관리자 조회에서는 품절 및 삭제 상태를 구분할 수 있는가?
- 비관적 읽기 잠금 조회가 정상적으로 실행되는가?

---

## 6.3 CartRepository

검증 항목:

- 회원별 장바구니를 조회할 수 있는가?
- 동일 장바구니와 메뉴 조합이 중복 저장되지 않는가?
- 장바구니를 비관적 쓰기 잠금으로 조회할 수 있는가?
- 장바구니 항목을 함께 조회할 수 있는가?

Unique 제약조건:

```text
(cart_id, menu_id)
```

---

## 6.4 PointRepository

검증 항목:

- 회원별 포인트를 조회할 수 있는가?
- 한 회원에게 포인트 행이 하나만 생성되는가?
- 포인트 행을 비관적 쓰기 잠금으로 조회할 수 있는가?
- 조건부 UPDATE 사용 시 잔액 부족 요청이 실패하는가?

조건부 UPDATE 예시:

```sql
UPDATE points
SET balance = balance - :amount
WHERE member_id = :memberId
  AND balance >= :amount
```

---

## 6.5 OrderRepository

검증 항목:

- 회원별 주문 내역을 조회할 수 있는가?
- 주문 상세와 주문 항목을 함께 조회할 수 있는가?
- 동일 회원과 동일 멱등키가 중복 저장되지 않는가?
- 최근 7일 주문 집계가 정상인가?
- 완료 주문만 인기 메뉴 집계에 포함되는가?

Unique 제약조건:

```text
(member_id, idempotency_key)
```

# 7. API 통합 테스트

통합 테스트는 Controller, Service, Repository 및 DB까지 연결해  
실제 요청 흐름을 검증한다.

Spring MVC API는 `MockMvc` 또는 `WebTestClient`를 사용한다.

---

## 7.1 인증 API

### 회원가입

검증 항목:

- 정상 회원가입
- 이메일 형식 오류
- 비밀번호 형식 오류
- 이메일 중복
- 필수값 누락

### 로그인

검증 항목:

- 정상 로그인
- 존재하지 않는 회원
- 비밀번호 불일치
- 탈퇴 회원 로그인
- Access Token과 Refresh Token 발급

### 로그아웃

검증 항목:

- Access Token Blacklist 등록
- Refresh Token 삭제
- 로그아웃 후 Access Token 사용 차단
- 로그아웃 후 Refresh Token 재발급 차단

### 토큰 재발급

검증 항목:

- 정상 재발급
- Refresh Token 없음
- 만료된 Refresh Token
- Redis에 등록되지 않은 Refresh Token
- 탈퇴 회원의 Refresh Token

---

## 7.2 메뉴 API

검증 항목:

- 메뉴 목록 조회
- 메뉴 상세 조회
- 존재하지 않는 메뉴 조회
- 판매중인 메뉴만 일반 사용자에게 노출
- 관리자 메뉴 등록
- 관리자 메뉴 수정
- 관리자 메뉴 상태 변경
- 관리자 메뉴 삭제
- 일반 사용자의 관리자 API 접근 차단

---

## 7.3 장바구니 API

검증 항목:

- 장바구니 조회
- 판매중 메뉴 추가
- 동일 메뉴 추가 시 수량 증가
- 수량 변경
- 항목 삭제
- 판매중이 아닌 메뉴 추가 실패
- 삭제된 메뉴 추가 실패
- 다른 회원 장바구니 접근 차단

---

## 7.4 포인트 API

검증 항목:

- 포인트 잔액 조회
- 정상 충전
- 0원 충전 실패
- 음수 충전 실패
- 포인트 이력 조회
- 다른 회원 포인트 접근 차단

---

## 7.5 주문 API

검증 항목:

- 정상 주문 및 결제
- 빈 장바구니 주문 실패
- 포인트 부족 주문 실패
- 판매중이 아닌 메뉴 주문 실패
- 삭제된 메뉴 주문 실패
- 존재하지 않는 메뉴 주문 실패
- 주문 총금액이 서버에서 계산되는지 확인
- 주문 완료 후 장바구니 항목 삭제
- 주문 완료 후 포인트 잔액 감소
- 주문 완료 후 결제 생성
- 주문 완료 후 포인트 이력 생성
- 주문 상세 조회
- 다른 회원 주문 접근 차단

---

## 7.6 인기 메뉴 API

검증 항목:

- 최근 7일 완료 주문만 포함
- 취소 주문 제외
- 7일 이전 주문 제외
- 수량이 아닌 주문 포함 횟수로 집계
- 주문 횟수 내림차순
- 횟수가 같으면 최근 주문 시각 내림차순
- 최근 주문 시각도 같으면 메뉴 ID 오름차순
- 상위 3개만 반환

예시:

```text
아메리카노가 한 주문에 5개 포함되어도 주문 횟수는 1회로 집계한다.
같은 주문에 동일 메뉴가 중복 행으로 존재하지 않도록 한다.
```

# 8. 트랜잭션 테스트

주문 처리 중 일부 단계에서 실패했을 때  
전체 트랜잭션이 Rollback되는지 검증한다.

## 8.1 포인트 차감 후 주문 저장 실패

예상 결과:

```text
포인트 잔액 원상복구
주문 미생성
주문 항목 미생성
결제 미생성
포인트 이력 미생성
장바구니 항목 유지
```

---

## 8.2 주문 저장 후 주문 항목 저장 실패

예상 결과:

```text
주문 Rollback
포인트 잔액 원상복구
결제 미생성
포인트 이력 미생성
장바구니 항목 유지
```

---

## 8.3 결제 저장 실패

```text
주문 Rollback

장바구니 항목 유지
```

---

## 8.4 포인트 이력 저장 실패

```text
주문 Rollback
주문 항목 Rollback
결제 Rollback
포인트 잔액 원상복구
장바구니 항목 유지
```

## 8.5 장바구니 삭제 실패

```text
주문 Rollback

```

## 8.6 Outbox Event 저장 실패

# 9. 동시성 테스트

동시성 테스트는 실제 여러 Thread에서 각각 별도의 트랜잭션으로 실행한다.

사용 도구:

```text
ExecutorService
CountDownLatch
CyclicBarrier
CompletableFuture
```

동시성 테스트 메서드 자체에 `@Transactional`을 적용하지 않는다.

테스트 메서드에 트랜잭션을 적용하면  
각 Thread의 Commit과 잠금 동작을 정확히 확인하기 어렵다.

---

## 9.1 동일 회원의 동시 주문

초기 상태:

```text
포인트 잔액: 10,000P
주문 A: 8,000P
주문 B: 7,000P
```

두 요청을 동시에 실행한다.

예상 결과:

```text
성공 주문: 1건
실패 주문: 1건
최종 포인트 잔액: 2,000P 또는 3,000P
포인트 음수 없음
주문과 결제 건수 일치
성공 주문 수와 포인트 이력 수 일치
```

정확히 어떤 주문이 먼저 성공하는지는 Thread 스케줄링에 따라 달라질 수 있으므로  
특정 주문의 성공 여부보다 최종 정합성을 검증한다.

---

## 9.2 여러 회원의 동시 주문

서로 다른 회원이 동시에 주문한다.

예상 결과:

- 각 회원의 포인트가 독립적으로 차감된다.
- 서로 다른 포인트 행 잠금으로 병렬 처리된다.
- 한 회원의 주문이 다른 회원의 주문을 불필요하게 차단하지 않는다.
- 모든 성공 주문의 주문, 결제, 포인트 이력이 일치한다.

---

## 9.3 포인트 충전과 주문 동시 실행

초기 상태:

```text
포인트 잔액: 5,000P
충전 요청: 5,000P
주문 요청: 8,000P
```

충전과 주문을 동시에 실행한다.

가능한 결과:

```text
충전이 먼저 Commit
→ 주문 성공
→ 최종 잔액 2,000P

주문이 먼저 실행
→ 포인트 부족으로 주문 실패
→ 충전 성공
→ 최종 잔액 10,000P
```

허용되지 않는 결과:

```text
포인트 음수
충전 금액 유실
주문 성공인데 포인트 미차감
포인트 차감인데 주문 미생성
```

---

## 9.4 주문과 장바구니 변경 동시 실행

주문 요청과 장바구니 수량 변경 요청을 동시에 실행한다.

예상 결과:

- 주문이 장바구니 쓰기 잠금을 획득하면 수정 요청은 대기한다.
- 주문 완료 후 장바구니 항목이 삭제된다.
- 주문 항목의 수량과 주문 금액이 일치한다.
- 주문 처리 중 장바구니 데이터가 중간 변경되지 않는다.

---

## 9.5 주문과 메뉴 수정 동시 실행

주문 요청과 관리자 메뉴 가격 변경을 동시에 실행한다.

예상 결과:

- 주문이 메뉴 읽기 잠금을 획득한 동안 관리자 수정은 대기한다.
- 주문에는 검증한 시점의 메뉴 가격이 저장된다.
- 주문 총금액과 주문 항목 가격이 일치한다.
- 가격 변경과 주문 데이터가 섞이지 않는다.

메뉴 상태 변경도 같은 방식으로 검증한다.

```text
ON_SALE → SOLD_OUT
ON_SALE → DELETED
```

---

## 9.6 동일 멱등키 동시 요청

동일 회원이 같은 `Idempotency-Key`로 주문 요청을 동시에 실행한다.

예상 결과:

```text
주문 1건
결제 1건
포인트 차감 1회
포인트 이력 1건
외부 전송 대상 1건
```

처리 방식:

```text
첫 번째 요청
→ 주문 생성 성공

두 번째 요청
→ 기존 주문 반환
또는
→ Unique 제약조건 충돌 후 기존 주문 조회
```

두 요청의 응답에는 같은 주문 ID가 반환되어야 한다.

---

## 9.7 다른 회원의 동일 멱등키

회원 A와 회원 B가 동일한 멱등키를 사용한다.

예상 결과:

- 회원별로 각각 주문을 생성할 수 있다.
- 멱등키는 회원 범위에서만 유일하다.

Unique 제약조건:

```text
(member_id, idempotency_key)
```

# 10. 동시성 전략 비교 테스트

포인트 차감 방식은 다음 전략을 비교할 수 있다.

```text
동시성 제어 없음
낙관적 락
비관적 락
조건부 UPDATE
```

## 10.1 동시성 제어 없음

검증 목적:

- Lost Update가 발생하는지 확인
- 포인트 초과 사용이 발생하는지 확인
- 정합성 문제가 실제로 재현되는지 확인

이 테스트는 문제 상황을 보여주기 위한 Baseline으로 사용한다.

---

## 10.2 낙관적 락

검증 항목:

- Version 충돌 발생 여부
- 충돌 요청 실패 수
- 재시도 횟수
- 재시도 후 성공률
- 높은 충돌 상황에서 성능 저하 여부

---

## 10.3 비관적 락

검증 항목:

- 초과 차감 방지
- 락 대기 시간
- 처리량
- Deadlock 발생 여부
- 같은 회원 요청이 순차적으로 처리되는지 확인

---

## 10.4 조건부 UPDATE

검증 항목:

- 잔액 부족 시 Update Count가 0인지 확인
- 포인트 초과 차감 방지
- 비관적 락 대비 처리량
- 주문 저장과 포인트 차감의 트랜잭션 정합성

---

## 10.5 비교 지표

| 지표 | 설명 |
|---|---|
| 성공 요청 수 | 정상 처리된 주문 수 |
| 실패 요청 수 | 포인트 부족, 충돌 및 시스템 오류 수 |
| 최종 잔액 | 기대값과 실제값 비교 |
| 평균 응답시간 | 전체 요청 평균 처리시간 |
| P95 | 상위 5% 구간 응답시간 |
| 처리량 | 초당 처리 요청 수 |
| 락 대기 시간 | 잠금 획득까지 대기한 시간 |
| 재시도 횟수 | 낙관적 락 충돌 후 재시도 횟수 |
| Deadlock 수 | Deadlock 발생 횟수 |
| 정합성 오류 수 | 주문, 결제, 포인트 불일치 수 |

# 11. 외부 Mock API 테스트

외부 Mock API는 실제 외부 시스템 대신 다음 상황을 재현한다.

```text
MockWebServer

```

## 11.1 정상 응답

```text
주문 Transaction Commit
```

- 올바른 주문 데이터가 전달되는가?

---

## 11.2 응답 지연

Mock API가 일정 시간 후 응답하도록 설정한다.

예시:

```text

5초
```

검증 항목:

- 외부 API 지연이 주문 API 응답시간에 반영되는가?
- DB 트랜잭션은 외부 호출 전에 Commit되는가?
- 지연 중 DB 락이 유지되지 않는가?
- 다른 주문 요청이 DB 락 때문에 불필요하게 대기하지 않는가?

## 11.3 HTTP 500

흐름:

```text
주문 Commit 성공

```

예상 결과:

```text
주문 상태 COMPLETED 유지

```

외부 API 실패로 주문을 Rollback하지 않는다.

## 11.4 Timeout

검증 항목:

- 주문 DB 데이터는 유지되는가?
- Timeout 로그가 기록되는가?
- 무한 대기가 발생하지 않는가?
- 재시도 정책이 있다면 설정 횟수만큼만 재시도하는가?

---

## 11.5 외부 API 호출 전 서버 종료

1차 구조의 한계를 검증하기 위한 시나리오다.

흐름:

```text
주문 Commit
→ 외부 API 호출 전 애플리케이션 종료
```

예상 결과:

- 주문은 DB에 존재한다.
- 외부 이벤트는 전송되지 않을 수 있다.
- 동기 전송 방식의 이벤트 유실 가능성을 확인한다.

이 결과를 2차 Outbox 구조와 비교한다.

# 12. Outbox 테스트

Transactional Outbox 저장과 조회는 #19에서 검증하고,
Kafka 발행 및 Consumer 처리는 #20에서 추가 검증한다.

## 12.1 주문과 Outbox 동시 저장

검증 항목:

- 주문과 Outbox Event가 같은 트랜잭션에서 저장되는가?
- 주문 성공 시 Outbox Event가 반드시 존재하는가?
- Outbox 저장 실패 시 주문도 Rollback되는가?

---

## 12.2 미발행 이벤트 조회

검증 항목:

- `PENDING` 상태 이벤트와 재시도 시각이 지난 `FAILED` 이벤트를 조회하는가?
- 생성 순서대로 조회하는가?
- 한 번에 처리할 개수를 제한하는가?
- 여러 Publisher가 같은 이벤트를 동시에 처리하지 않는가?

---

## 12.3 Kafka 발행 성공

예상 흐름:

```text
Outbox PENDING
→ Kafka 발행 성공
→ Outbox PUBLISHED
```

검증 항목:

- Topic에 이벤트가 발행되는가?
- Event ID가 유지되는가?
- Outbox 상태가 PUBLISHED로 변경되는가?
- 발행 시각이 기록되는가?

---

## 12.4 Kafka 발행 실패

예상 흐름:

```text
Outbox PENDING
→ Kafka 발행 실패
→ PENDING 또는 FAILED 유지
→ 재시도
```

검증 항목:

- 실패 이벤트가 삭제되지 않는가?
- 재시도 횟수가 증가하는가?
- `last_error`와 `next_retry_at`이 기록되는가?
- 다음 Publisher 실행에서 다시 조회되는가?
- 최대 재시도 초과 시 자동 재시도 대상에서 제외되고 수동 재처리 대상으로 남는가?

---

## 12.5 Publisher 중복 실행

같은 Outbox Event를 Publisher가 두 번 처리하는 상황을 재현한다.

검증 항목:

- 동일 Event ID가 유지되는가?
- Publisher가 조회한 이벤트를 lease로 예약해 동시에 같은 이벤트를 선택하지 않도록 하는가?
- Kafka에 중복 발행될 수 있음을 고려하는가?
- Consumer에서 중복 처리를 방지하는가?

Outbox는 이벤트 유실을 방지하지만  
전송 횟수를 정확히 한 번으로 완전히 보장하지는 않는다.

따라서 Consumer 멱등성이 필요하다.

---

## 12.6 Outbox 운영 조회 API

관리자가 Outbox Event의 발행 상태와 재시도 정보를 조회할 수 있는지 검증한다.

검증 항목:

- 관리자만 Outbox 이벤트를 조회할 수 있는가?
- `PENDING`, `FAILED`, `PUBLISHED` 상태별 필터링이 가능한가?
- `retry_count`, `next_retry_at`, `last_error`가 응답에 포함되는가?
- 페이징과 정렬 요청이 공통 페이지 응답 형식으로 반환되는가?
- 잘못된 상태 또는 페이징 요청은 공통 오류 응답으로 반환되는가?

# 13. Kafka Consumer 테스트

## 13.1 정상 이벤트 처리

흐름:

```text
Kafka Order Event
→ Consumer
→ 외부 데이터 플랫폼 호출
→ 처리 완료
```

검증 항목:

- 올바른 Topic을 구독하는가?
- Event Payload가 정상 역직렬화되는가?
- 외부 API가 한 번 호출되는가?
- 처리 완료 상태가 기록되는가?

---

## 13.2 동일 이벤트 중복 수신

동일한 `eventId` 이벤트를 두 번 발행한다.

예상 결과:

```text
첫 번째 이벤트 처리
→ 외부 API 호출

두 번째 이벤트 수신
→ 이미 처리된 Event ID 확인
→ 외부 API 호출 생략
```

검증 항목:

- 중복 이벤트가 한 번만 처리되는가?
- 처리 완료 Event ID가 저장되는가?
- `processed_kafka_events`의 `COMPLETED` 상태를 기준으로 재호출을 생략하는가?
- 동일 주문이 외부 시스템에 중복 반영되지 않는가?

## 13.2.1 처리 중 이벤트 lease 만료

`PROCESSING` 저장 후 Consumer가 종료되어 `complete()` 또는 `fail()`이 실행되지 않은 상황을 재현한다.

검증 항목:

- `processing_deadline_at`이 유효한 `PROCESSING` 이벤트는 중복 처리로 보고 건너뛰는가?
- `processing_deadline_at`이 지난 `PROCESSING` 이벤트는 재처리하는가?
- 재처리 후 외부 API가 호출되고 `COMPLETED` 또는 `FAILED`로 전이되는가?

---

## 13.3 Consumer 처리 실패

검증 항목:

- 외부 API 500 발생 시 재시도하는가?
- 재시도 횟수 제한이 적용되는가?
- 재시도 중 Offset Commit 시점이 올바른가?
- 최대 재시도 후 Dead Letter Topic으로 이동하는가?

---

## 13.4 Dead Letter Topic

검증 항목:

# 16. 테스트 데이터 관리

테스트는 서로 독립적으로 실행되어야 한다.

- 테스트 실행 순서에 의존하지 않는다.
- 다른 테스트가 생성한 데이터를 사용하지 않는다.
- 테스트마다 필요한 데이터를 직접 생성한다.
- 고정된 운영 데이터에 의존하지 않는다.
- 테스트 후 데이터가 정리되도록 한다.
- 동시성 테스트는 테스트 전후 DB 상태를 직접 초기화한다.

테스트 Fixture가 반복되면 다음과 같이 분리할 수 있다.

```text
MemberFixture
MenuFixture
CartFixture
PointFixture
OrderFixture
```

예시:

```java
Member member = MemberFixture.createMember();
Menu menu = MenuFixture.createAmericano();
Point point = PointFixture.createPoint(member, 10_000L);
```

# 18. 테스트 완료 기준

다음 조건을 만족하면 1차 구현 테스트가 완료된 것으로 본다.

- 핵심 API 정상 및 실패 시나리오가 통과한다.
- 포인트 차감과 주문 저장의 Rollback이 검증된다.
- 동일 회원 동시 주문에서 포인트가 음수가 되지 않는다.
- 동일 멱등키 요청으로 주문이 한 건만 생성된다.
- 주문, 주문 항목, 결제 및 포인트 이력 건수가 일치한다.
- 외부 Mock API 실패에도 주문 데이터가 유지된다.
- 인기 메뉴 집계 기준이 비즈니스 규칙과 일치한다.
- 다중 인스턴스 환경에서도 DB와 Redis 기준으로 동일하게 동작한다.

2차 고도화는 다음 조건을 추가로 만족해야 한다.

- 주문과 Outbox Event가 함께 저장된다.
- Kafka 장애 시 이벤트가 Outbox에 유지된다.
- Kafka 복구 후 이벤트가 재발행된다.
- 동일 Event ID가 중복 처리되지 않는다.
- Consumer 최종 실패 이벤트가 Dead Letter Topic으로 이동한다.
- 외부 시스템 장애가 주문 API 응답시간에 직접 영향을 주지 않는다.

# 20. 최종 요약

- 핵심 도메인 규칙은 단위 테스트로 검증한다.
- Repository와 잠금 테스트는 실제 MySQL 환경에서 실행한다.
- Controller부터 DB까지 통합 테스트를 작성한다.
- 주문의 각 저장 단계 실패 시 전체 Rollback을 검증한다.
- 동시성 테스트는 실제 여러 Thread와 별도 트랜잭션으로 실행한다.
- 동일 회원 주문, 포인트 충전, 장바구니 변경 및 메뉴 수정을 동시에 검증한다.
- 동일 멱등키 요청은 주문, 결제 및 포인트 차감이 한 번만 발생해야 한다.
- 외부 Mock API 지연과 장애가 주문 데이터에 미치는 영향을 검증한다.
- Outbox와 Kafka 구조에서는 이벤트 유실과 중복 처리를 검증한다.
- 성능 테스트에서는 정합성과 응답시간을 함께 비교한다.
