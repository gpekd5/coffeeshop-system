# Business Rules

이 문서는 커피 주문 및 포인트 결제 시스템의 비즈니스 정책을 정의한다.

구체적인 API 구조, 데이터베이스 설계, 트랜잭션 범위,
동시성 제어 기술 및 인프라 구성은 별도의 설계 문서에서 정의한다.

---

# 1. 핵심 설계 목표

이 프로젝트는 단순 CRUD 구현보다 다음 문제 해결을 우선한다.

1. 동시에 여러 주문이 발생해도 포인트가 초과 사용되지 않도록 한다.
2. 주문, 주문 항목, 포인트 차감 및 포인트 이력의 데이터 정합성을 보장한다.
3. 여러 애플리케이션 인스턴스에서 요청을 처리해도 동일한 결과를 보장한다.
4. 동일한 요청이 반복되어도 중복 주문이나 중복 결제가 발생하지 않도록 한다.
5. 외부 데이터 수집 플랫폼에 장애가 발생해도 완료된 주문 정보가 유실되지 않도록 한다.
6. 동시성 테스트와 부하 테스트를 통해 정합성과 처리 성능을 검증한다.
7. 주문 진입 방식과 결제 수단이 추가되더라도 핵심 주문 규칙을 재사용할 수 있도록 한다.

---

# 2. 공통 정책

## 2.1 식별자

- 회원, 메뉴, 장바구니 항목, 주문, 주문 항목, 포인트 이력은 각각 고유 식별자를 가진다.
- 인증이 필요한 기능에서는 서버 인증 컨텍스트의 회원 식별자를 사용한다.
- Request Body 또는 Query Parameter로 전달된 회원 식별자를 인증된 사용자로 신뢰하지 않는다.
- 외부 데이터 전송 이벤트는 중복 처리를 방지할 수 있도록 고유 식별자를 가진다.

## 2.2 날짜와 시간

- 주요 데이터는 생성 일시와 수정 일시를 관리한다.
- 서버 내부의 시간 기준은 하나로 통일한다.
- 사용자에게 반환하는 날짜와 시간 형식은 API 명세에서 정의한다.
- 인기 메뉴 집계 기준은 조회 시점을 기준으로 연속된 최근 7일이다.

## 2.3 금액과 포인트

- 모든 금액은 원 단위의 정수로 관리한다.
- 소수점 금액은 지원하지 않는다.
- 포인트는 정수로 관리한다.
- 1원은 1P로 환산한다.
- 금액과 포인트는 음수가 될 수 없다.
- 가격과 최종 결제 금액은 서버에서 계산하고 검증한다.
- 클라이언트가 전달한 메뉴 가격이나 총 결제 금액을 신뢰하지 않는다.

## 2.4 공통 API 응답

- 모든 API는 프로젝트에서 정의한 공통 응답 형식을 따른다.
- 정상 응답과 오류 응답은 일관된 필드 구조를 사용한다.
- 입력값 오류, 인증 실패, 권한 부족, 리소스 없음, 포인트 부족 등을 구분된 오류 코드로 관리한다.
- 내부 서버 오류의 상세 원인은 사용자에게 직접 노출하지 않는다.

## 2.5 Spring Security 오류 응답

- Spring Security Filter Chain에서 발생한 인증 및 인가 오류도 공통 오류 응답 형식을 따른다.
- 인증되지 않은 사용자의 접근은 `401 Unauthorized`로 반환한다.
- 권한이 부족한 사용자의 접근은 `403 Forbidden`으로 반환한다.
- 인증 및 인가 오류에도 프로젝트에서 정의한 오류 코드를 사용한다.
- 오류 응답에 토큰 원문이나 내부 예외 정보를 노출하지 않는다.

# 3. 회원 정책

## 3.1 회원가입

- 사용자는 이메일, 비밀번호, 이름을 입력하여 가입한다.
- 로그인 ID는 이메일 주소를 사용한다.
- 이메일은 올바른 이메일 형식이어야 한다.
- 이메일의 앞뒤 공백을 제거한다.
- 이메일은 소문자로 정규화하여 저장한다.
- 정규화된 이메일을 기준으로 중복 여부를 확인한다.
- 동일한 이메일로 중복 가입할 수 없다.
- 비밀번호는 평문으로 저장하지 않는다.
- 비밀번호는 단방향 암호화하여 저장한다.
- 회원가입 시 기본 권한은 `USER`이다.
- 회원가입 시 포인트 잔액은 0P로 생성한다.

## 3.2 회원 권한

회원 권한은 다음과 같이 구분한다.

- `USER`: 일반 사용자
- `ADMIN`: 관리자

일반 사용자는 다음 기능을 사용할 수 있다.

- 메뉴 조회
- 장바구니 관리
- 포인트 충전 및 조회
- 주문 및 결제
- 본인의 포인트 및 주문 이력 조회

관리자는 일반 사용자 기능과 함께 다음 기능을 사용할 수 있다.

- 메뉴 등록
- 메뉴 수정
- 메뉴 판매 상태 변경
- 메뉴 삭제
- 전체 주문 조회

## 3.3 회원 상태

회원 상태는 다음과 같이 구분한다.

- `ACTIVE`: 정상 이용 가능
- `INACTIVE`: 이용 제한
- `WITHDRAWN`: 탈퇴

초기 회원 상태는 `ACTIVE`이다.

## 3.4 회원 삭제

- 회원 데이터는 주문 및 포인트 이력 보존을 위해 물리 삭제하지 않는다.
- 회원 탈퇴는 상태 변경 또는 Soft Delete 방식으로 처리한다.
- 탈퇴한 회원의 과거 주문 및 포인트 이력은 유지한다.

# 4. 인증 및 인가 정책

## 4.1 로그인

- 사용자는 이메일과 비밀번호로 로그인한다.
- 이메일 또는 비밀번호가 일치하지 않으면 로그인에 실패한다.
- 비활성화되거나 탈퇴한 사용자는 로그인할 수 없다.
- 로그인 성공 시 Access Token과 Refresh Token을 발급한다.

## 4.2 Access Token

- Access Token은 비교적 짧은 유효기간을 가진다.
- 인증이 필요한 요청에서 Access Token으로 사용자를 식별한다.
- 로그아웃한 Access Token은 남은 유효기간 동안 Redis Blacklist에 저장한다.
- 인증 처리 시 Access Token이 Blacklist에 등록되어 있는지 확인한다.
- Blacklist 데이터의 TTL은 Access Token의 남은 유효기간과 동일하게 설정한다.

## 4.3 Refresh Token

- Refresh Token은 Redis Whitelist에 저장한다.
- Access Token 재발급 시 전달받은 Refresh Token이 Whitelist에 존재하는지 확인한다.
- Redis에 저장되지 않았거나 만료된 Refresh Token은 사용할 수 없다.
- 로그아웃 시 해당 회원의 Refresh Token을 Redis에서 삭제한다.
- Refresh Token 교체 정책은 보안 설계 문서에서 정의한다.

## 4.4 리소스 접근 제어

- 사용자는 본인의 장바구니, 포인트, 포인트 이력 및 주문 내역만 조회할 수 있다.
- 다른 회원의 식별자를 임의로 전달하여 다른 회원의 리소스에 접근할 수 없다.
- 관리자 기능은 `ADMIN` 권한을 가진 사용자만 수행할 수 있다.

# 5. 삭제 정책

## 5.1 Soft Delete 적용 대상

다음 데이터는 이력 및 참조 관계 보존을 위해 물리 삭제하지 않는다.

- 회원
- 메뉴
- 주문
- 주문 항목
- 포인트 이력

Soft Delete된 데이터는 일반 조회 대상에서 제외한다.

과거 주문 및 관리자 조회처럼 이력 확인이 필요한 경우에는 조회할 수 있어야 한다.

## 5.2 Hard Delete 적용 대상

장바구니와 장바구니 항목은 주문 전 임시 데이터로 취급한다.

- 장바구니 항목 삭제 시 실제 데이터를 삭제한다.
- 장바구니 전체 비우기 시 관련 항목을 실제로 삭제한다.
- 주문 완료 시 주문에 포함된 장바구니 항목을 실제로 삭제한다.
- 장바구니 데이터는 주문 이력의 근거로 사용하지 않는다.
- 주문 이력에 필요한 메뉴 이름, 가격 및 수량은 주문 항목에 별도로 저장한다.

# 6. 메뉴 정책

## 6.1 메뉴 정보

메뉴는 다음 정보를 가진다.

- 메뉴 이름
- 메뉴 설명
- 카테고리
- 가격
- 판매 상태
- 생성 일시
- 수정 일시
- 삭제 여부 또는 삭제 일시

## 6.2 메뉴 카테고리

메뉴는 하나의 카테고리에 속한다.

- `COFFEE`: 커피
- `NON_COFFEE`: 논커피 음료
- `DECAF`: 디카페인 음료
- `BAKERY`: 빵 및 샌드위치
- `ETC`: 기타

정책은 다음과 같다.

- 메뉴 등록 시 카테고리는 필수값이다.
- 관리자는 메뉴 카테고리를 변경할 수 있다.
- 사용자는 카테고리별로 메뉴 목록을 조회할 수 있다.
- 카테고리를 동적으로 생성·수정·삭제하는 기능은 초기 구현 범위에서 제외한다.

## 6.3 메뉴 가격

- 메뉴 가격은 0원보다 커야 한다.
- 메뉴 가격은 원 단위 정수로 관리한다.
- 가격 변경은 이후 주문부터 적용한다.
- 이미 완료된 주문에는 변경된 메뉴 가격이 영향을 주지 않는다.

## 6.4 메뉴 판매 상태

메뉴 판매 상태는 다음과 같이 구분한다.

- `ON_SALE`: 판매 중
- `SOLD_OUT`: 일시 품절
- `INACTIVE`: 판매 중지

판매 중인 메뉴만 장바구니에 추가하고 주문할 수 있다.

품절, 판매 중지 또는 삭제된 메뉴는 주문할 수 없다.

## 6.5 메뉴 관리

- 메뉴 등록, 수정 및 삭제는 관리자만 수행할 수 있다.
- 메뉴 이름과 카테고리, 가격은 필수값이다.
- 메뉴의 이름, 설명, 카테고리, 가격 및 판매 상태를 변경할 수 있다.
- 메뉴 삭제는 Soft Delete 방식으로 처리한다.
- 메뉴 변경 및 삭제는 과거 주문 정보에 영향을 주지 않는다.

# 7. 장바구니 정책

## 7.1 장바구니 소유

- 장바구니는 로그인한 사용자만 사용할 수 있다.
- 사용자는 본인의 장바구니만 조회하고 변경할 수 있다.
- 사용자별로 하나의 활성 장바구니를 사용한다.
- 활성 장바구니가 없으면 최초 장바구니 접근 시 생성한다.

## 7.2 메뉴 추가

- 판매 중인 메뉴만 장바구니에 추가할 수 있다.
- 요청 수량은 1개 이상이어야 한다.
- 이미 장바구니에 있는 메뉴를 다시 추가하면 기존 수량에 요청 수량을 더한다.
- 메뉴별 최대 주문 수량은 별도 정책으로 결정한다.

## 7.3 수량 변경

- 장바구니 항목의 수량은 1개 이상이어야 한다.
- 수량을 0으로 변경하는 요청은 허용하지 않는다.
- 항목 제거는 삭제 기능을 통해 처리한다.

## 7.4 장바구니 가격

- 장바구니에서는 현재 메뉴 가격을 기준으로 예상 결제 금액을 제공한다.
- 장바구니에 담은 시점의 가격은 최종 결제 가격으로 보장하지 않는다.
- 주문 요청 시 서버가 메뉴 가격과 판매 상태를 다시 확인한다.
- 장바구니에 담은 이후 가격이 변경된 경우 변경된 가격을 적용한다.

## 7.5 장바구니 주문

- 현재 모든 사용자 주문은 장바구니를 기준으로 생성한다.
- 장바구니가 비어 있으면 주문할 수 없다.
- 사용자는 본인의 장바구니만 주문할 수 있다.
- 주문 시 클라이언트가 가격이나 총 결제 금액을 전달하지 않는다.
- 장바구니 항목 중 하나라도 주문할 수 없는 상태라면 전체 주문을 실패 처리한다.
- 주문 성공 시 장바구니 항목을 Hard Delete한다.
- 주문 실패 시 장바구니 항목을 유지한다.

# 8. 주문 처리 확장 정책

## 8.1 현재 주문 방식

현재 사용자는 다음 흐름으로 주문한다.

`메뉴 선택 → 장바구니 추가 → 장바구니 확인 → 주문 및 결제`

## 8.2 향후 주문 진입 방식

향후 다음 주문 방식이 추가될 수 있다.

- 모바일 주문
- 키오스크 주문
- 관리자 대리 주문
- 예약 주문
- 외부 채널 주문

## 8.3 공통 주문 처리

- 주문 진입 방식은 주문할 메뉴와 수량을 구성하는 책임을 가진다.
- 메뉴 검증, 가격 계산, 결제, 주문 저장 및 외부 전송 이벤트 생성은 공통 주문 처리 규칙을 사용한다.
- 새로운 주문 진입 방식이 추가되어도 주문 핵심 로직을 중복 구현하지 않는다.
- 현재 장바구니 주문도 공통 주문 처리 절차를 사용한다.

공통 주문 처리 흐름은 다음과 같다.

1. 주문 대상 확인
2. 메뉴 존재 여부 확인
3. 판매 상태 확인
4. 현재 가격 조회
5. 총 주문 금액 계산
6. 결제 수행
7. 주문 및 주문 항목 저장
8. 결제 및 포인트 이력 저장
9. 외부 데이터 전송 이벤트 생성
10. 주문 진입 데이터 후처리

# 9. 포인트 정책

## 9.1 포인트 계정

- 회원은 하나의 포인트 잔액을 가진다.
- 회원가입 시 포인트 잔액은 0P로 시작한다.
- 포인트 잔액은 음수가 될 수 없다.

## 9.2 포인트 충전

- 로그인한 사용자는 포인트를 충전할 수 있다.
- 충전 금액은 0원보다 커야 한다.
- 1원당 1P의 비율로 포인트를 충전한다.
- 기본 과제 범위에서는 요청 금액만큼 포인트를 증가시킨다.
- 충전 완료 시 포인트 충전 이력을 저장한다.
- 포인트 증가와 충전 이력 저장 중 일부만 성공하는 상태를 허용하지 않는다.

## 9.3 포인트 사용

- 초기 주문 결제 수단은 포인트만 지원한다.
- 주문 금액만큼 사용자의 포인트를 차감한다.
- 보유 포인트가 주문 금액보다 적으면 주문할 수 없다.
- 포인트 차감 완료 시 포인트 사용 이력을 저장한다.
- 포인트 차감과 사용 이력 저장 중 일부만 성공하는 상태를 허용하지 않는다.

## 9.4 포인트 이력

포인트 이력 유형은 다음과 같이 구분한다.

- `CHARGE`: 포인트 충전
- `USE`: 주문 결제
- `CANCEL`: 충전 또는 결제 취소
- `ADJUST`: 관리자 또는 시스템 조정

초기 구현에서는 `CHARGE`와 `USE`를 필수로 구현한다.

포인트 이력에는 다음 정보를 저장한다.

- 사용자
- 이력 유형
- 변동 포인트
- 변동 후 잔액
- 관련 주문 또는 결제 식별자
- 발생 일시

## 9.5 포인트 정합성

- 동시에 여러 주문이 요청되어도 보유 포인트를 초과하여 사용할 수 없다.
- 포인트 충전과 주문이 동시에 발생해도 최종 잔액이 정확해야 한다.
- 포인트 차감에 실패한 주문은 완료 처리하지 않는다.
- 주문 저장에 실패하면 포인트 차감이 완료 상태로 남아서는 안 된다.
- 포인트 잔액, 포인트 이력 및 주문 결제 금액은 서로 일치해야 한다.

# 10. 결제 확장 정책

## 10.1 초기 결제 수단

- 초기 결제 수단은 포인트이다.
- 주문은 포인트 결제의 내부 구현 세부사항에 직접 의존하지 않도록 책임을 분리한다.

## 10.2 향후 결제 수단

향후 다음 결제 수단이 추가될 수 있다.

- 카드 결제
- 간편결제
- 기프트카드
- 쿠폰
- 포인트와 외부 결제의 복합결제

## 10.3 공통 결제 책임

각 결제 수단은 다음 책임을 가진다.

- 결제 가능 여부 확인
- 결제 금액 검증
- 결제 수행
- 결제 성공 또는 실패 결과 반환
- 중복 결제 방지
- 결제 이력 생성에 필요한 정보 제공

결제 수단이 추가되어도 주문 생성 및 주문 항목 저장 규칙은 동일하게 유지한다.

# 11. 주문 정책

## 11.1 주문 가능 조건

다음 조건을 모두 만족해야 주문할 수 있다.

- 로그인한 사용자일 것
- 장바구니가 비어 있지 않을 것
- 모든 메뉴가 존재할 것
- 모든 메뉴가 판매 중일 것
- 모든 수량이 1개 이상일 것
- 사용자의 포인트가 총 주문 금액 이상일 것

## 11.2 주문 금액 계산

- 주문 금액은 서버가 주문 시점의 메뉴 가격과 수량을 기준으로 계산한다.
- 총 주문 금액은 각 주문 항목의 `주문 당시 가격 × 수량` 합계이다.
- 클라이언트가 전달한 가격과 총 결제 금액을 신뢰하지 않는다.

## 11.3 주문 당시 정보 보존

주문 항목에는 다음 정보를 저장한다.

- 메뉴 식별자
- 주문 당시 메뉴 이름
- 주문 당시 메뉴 카테고리
- 주문 당시 메뉴 가격
- 주문 수량
- 항목별 결제 금액

메뉴가 변경되거나 삭제되어도 과거 주문 내용은 유지되어야 한다.

## 11.4 주문 상태

주문 상태는 다음과 같이 구분한다.

- `COMPLETED`: 결제와 주문 저장이 완료된 상태
- `CANCELLED`: 주문이 취소된 상태
- `FAILED`: 주문 처리에 실패한 상태

초기 구현에서는 `COMPLETED` 상태를 중심으로 구현한다.

## 11.5 주문 데이터 정합성

다음 데이터는 하나의 논리적인 작업 단위로 처리한다.

- 주문 저장
- 주문 항목 저장
- 포인트 차감
- 포인트 사용 이력 저장
- 외부 전송 이벤트 생성
- 주문 성공 후 장바구니 항목 삭제

다음 상태는 허용하지 않는다.

- 포인트만 차감되고 주문이 없는 상태
- 주문만 생성되고 포인트가 차감되지 않은 상태
- 주문 금액과 포인트 사용 금액이 다른 상태
- 주문은 완료됐지만 포인트 이력이 없는 상태
- 주문은 완료됐지만 외부 전송 대상 이벤트가 생성되지 않은 상태

## 11.6 중복 주문

- 네트워크 재시도 또는 반복 클릭으로 동일 요청이 중복될 수 있음을 고려한다.
- 동일한 주문 요청이 여러 번 전달되어도 중복 주문이나 중복 결제가 발생하지 않아야 한다.
- 구체적인 멱등키 생성 및 전달 방식은 API와 아키텍처 문서에서 정의한다.

# 12. 외부 데이터 수집 플랫폼 정책

## 12.1 목적

- 완료된 주문 정보는 분석, 통계 및 추천 데이터 활용을 위해 외부 데이터 수집 플랫폼으로 전송한다.
- 외부 데이터 수집 플랫폼은 인기 메뉴 조회의 필수 원본 데이터로 사용하지 않는다.
- 외부 전송 실패나 지연이 인기 메뉴 조회 결과에 영향을 주어서는 안 된다.

## 12.2 전송 대상

결제가 완료된 주문만 전송한다.

최소 전송 정보는 다음과 같다.

- 이벤트 식별자
- 주문 식별자
- 사용자 식별자
- 메뉴 식별자
- 주문 수량
- 결제 금액
- 주문 일시

## 12.3 전송 시점과 실패

- 주문 완료 후 가능한 빠르게 전송한다.
- 주문 데이터와 외부 전송 대상 정보가 유실되지 않아야 한다.
- 외부 플랫폼 장애가 이미 완료된 주문을 실패 상태로 변경하지 않는다.
- 실패한 전송은 재시도할 수 있어야 한다.
- 실패 원인과 재시도 횟수를 확인할 수 있어야 한다.

## 12.4 중복 전송

- 동일한 이벤트가 여러 번 전송될 수 있음을 고려한다.
- 전송 데이터에 고유 이벤트 식별자를 포함한다.
- 동일 이벤트가 반복 전달되어도 중복 집계되지 않도록 한다.

## 12.5 단계별 이벤트 처리 고도화

### 1차 구현

- 주문 및 포인트 데이터 정합성 검증을 우선한다.
- 주문 완료 후 외부 데이터 수집 플랫폼의 Mock API를 동기 방식으로 호출한다.
- 외부 시스템 지연과 장애가 주문 응답시간 및 성공률에 미치는 영향을 측정한다.
- 인기 메뉴는 내부 데이터베이스의 완료 주문 데이터를 기준으로 집계한다.

### 2차 고도화

- 주문 완료 시 주문 데이터와 함께 Outbox Event를 동일한 데이터베이스 트랜잭션에 저장한다.
- 별도의 Publisher가 커밋된 Outbox Event를 Kafka의 주문 완료 Topic으로 발행한다.
- Kafka Consumer는 주문 완료 이벤트를 받아 데이터 수집 및 후속 통계 처리를 수행한다.
- Kafka 발행 또는 Consumer 처리 실패가 완료된 주문 상태를 변경하지 않도록 한다.
- 발행 실패 이벤트는 재시도할 수 있어야 한다.
- Producer와 Consumer는 이벤트 식별자를 기준으로 중복 발행 및 중복 소비에 대응한다.
- 초기 인기 메뉴 조회는 내부 완료 주문 데이터를 기준으로 유지한다.
- 향후 Kafka Consumer를 이용한 별도 인기 메뉴 집계 저장소 도입 여부를 비교 검토한다.

# 13. 인기 메뉴 정책

## 13.1 명칭

- 사용자 화면에는 `금주의 인기 메뉴`라는 이름으로 표시할 수 있다.
- 실제 집계 기간은 달력상의 이번 주가 아니라 조회 시점을 기준으로 최근 7일이다.

## 13.2 집계 원본

- 인기 메뉴는 내부 시스템의 완료 주문 데이터를 기준으로 집계한다.
- 외부 데이터 수집 플랫폼의 전송 결과에 의존하지 않는다.

## 13.3 집계 대상

- 결제가 완료된 주문만 집계한다.
- 실패한 주문은 집계하지 않는다.
- 취소 기능이 추가되면 취소된 주문은 집계에서 제외한다.
- 현재 판매 중지되거나 삭제된 메뉴라도 집계 기간 내 완료 주문이 있다면 집계 대상에 포함한다.

## 13.4 인기 기준

- 과제 원문의 메뉴별 주문 횟수 요구사항을 기준으로 한다.
- 하나의 주문에 특정 메뉴가 포함되면 해당 메뉴의 주문 횟수를 1회 증가시킨다.
- 동일 주문에서 같은 메뉴를 여러 개 주문해도 주문 횟수는 1회로 계산한다.
- 주문 수량 기반 통계는 별도 통계 기능으로 구분한다.

## 13.5 조회 개수와 정렬

- 인기 메뉴 상위 3개를 반환한다.
- 집계 대상 메뉴가 3개보다 적으면 존재하는 메뉴만 반환한다.
- 집계 대상 주문이 없으면 빈 목록을 반환한다.

주문 횟수가 같으면 다음 순서로 정렬한다.

1. 주문 횟수 내림차순
2. 최근 주문 시각 내림차순
3. 메뉴 식별자 오름차순

## 13.6 정확성

- 인기 메뉴별 주문 횟수는 실제 완료 주문 데이터와 일치해야 한다.
- 초기 구현에서는 성능보다 집계 정확성을 우선한다.
- 트래픽 증가로 별도 집계 저장소를 도입하더라도 원본 주문 데이터와의 일관성을 고려한다.

# 14. 동시 요청 및 다중 인스턴스 정책

## 14.1 동일 사용자 동시 주문

사용자가 5,000P를 보유하고 있고 3,000P 주문 요청이 동시에 2건 발생하면 다음 결과가 되어야 한다.

- 주문 1건 성공
- 주문 1건 실패
- 최종 포인트 2,000P

다음 결과는 허용하지 않는다.

- 주문 2건 모두 성공
- 최종 포인트 음수
- 성공 주문 수와 포인트 사용 이력 수 불일치

## 14.2 충전과 주문 동시 발생

충전과 주문이 동시에 처리되더라도 다음 식이 성립해야 한다.

`최종 포인트 = 초기 포인트 + 성공한 충전 금액 - 성공한 주문 금액`

## 14.3 다중 서버 및 다중 인스턴스

- 애플리케이션 인스턴스가 달라도 동일한 결과를 보장한다.
- 인증 상태, 포인트 잔액, 주문 상태 및 외부 전송 상태를 서버 로컬 메모리에만 저장하지 않는다.
- 로그아웃 및 토큰 상태는 모든 인스턴스가 공유하는 Redis에서 확인한다.
- 포인트와 주문 데이터는 모든 인스턴스가 공유하는 데이터베이스를 기준으로 처리한다.
- 단일 애플리케이션 인스턴스 내부의 락만으로 정합성을 보장하지 않는다.
- 구체적인 동시성 제어 방식은 아키텍처 문서에서 결정한다.

# 15. 부하 테스트 정책

## 15.1 목표

부하 테스트는 단순 응답 속도 측정보다 동시 요청 상황에서 데이터 정합성을 유지하며 처리량을 확보하는 것을 목표로 한다.

## 15.2 동일 사용자 동시 주문

예시 조건은 다음과 같다.

- 초기 포인트: 100,000P
- 메뉴 가격: 1,000P
- 동시 주문 요청: 200건

기대 결과는 다음과 같다.

- 성공 주문: 100건
- 실패 주문: 100건
- 최종 포인트: 0P
- 완료 주문: 100건
- 포인트 사용 이력: 100건

## 15.3 다중 사용자 주문 부하

- 여러 사용자가 동시에 주문하는 상황을 검증한다.
- 동시 사용자 수를 점진적으로 증가시킨다.
- 평균 응답시간, P95, 처리량 및 실패율을 측정한다.
- DB Connection Pool 고갈, Lock 대기, Deadlock 및 Timeout 여부를 확인한다.

## 15.4 충전과 주문 동시 발생

- 동일 회원의 포인트 충전과 주문을 동시에 요청한다.
- 성공한 충전과 주문이 최종 포인트 잔액에 정확히 반영되는지 확인한다.

## 15.5 동시성 제어 방식 비교

필요한 경우 다음 방식을 비교한다.

- 동시성 제어 없음
- DB 비관적 락
- 조건부 UPDATE
- Redis 분산 락

비교 항목은 다음과 같다.

- 데이터 정합성
- 성공 및 실패 요청 수
- 평균 응답시간
- P95 응답시간
- 처리량
- Lock 대기 시간
- Deadlock 및 Timeout 여부

## 15.6 외부 데이터 전송 방식 비교

외부 데이터 전송 방식은 다음 두 단계로 비교한다.

### v1: 동기 Mock API 전송

- 주문 처리 흐름 안에서 외부 Mock API를 호출한다.
- 외부 API에 의도적인 지연과 실패를 발생시킨다.
- 외부 시스템 장애가 주문 응답시간과 주문 성공률에 미치는 영향을 측정한다.

### v2: Outbox + Kafka 비동기 전송

- 주문 처리 트랜잭션에서 Outbox Event를 저장한다.
- 주문 응답과 Kafka 발행 및 Consumer 처리를 분리한다.
- 외부 시스템 장애 상황에서도 주문이 정상적으로 완료되는지 확인한다.
- Outbox Event와 Kafka Event의 유실 및 중복 처리 여부를 검증한다.

비교 항목은 다음과 같다.

- 평균 응답시간
- P95 응답시간
- 초당 처리량
- 주문 실패율
- 외부 시스템 장애 시 주문 성공률
- 이벤트 유실 여부
- 이벤트 중복 처리 여부
- Outbox 적체량
- Kafka Consumer Lag

동시성 제어 방식 비교와 외부 데이터 전송 방식 비교는 별도의 테스트로 수행한다.

한 테스트에서 동시성 제어 방식과 이벤트 전송 방식을 동시에 변경하지 않는다.

# 16. 확장성 설계 원칙

- 현재 장바구니 주문과 향후 모바일, 키오스크 등의 주문 진입 방식을 공통 주문 처리 절차로 연결한다.
- 주문, 결제, 포인트 및 외부 데이터 전송의 책임을 분리한다.
- 결제 수단이 추가되어도 주문 생성 및 주문 항목 저장 규칙을 재사용한다.
- 주문 항목에는 주문 당시 메뉴 정보를 저장하여 메뉴 변경 및 삭제와 독립적으로 주문 이력을 보존한다.
- 포인트 잔액과 변경 이력을 함께 관리하여 충전, 사용, 취소 및 조정 기능으로 확장할 수 있도록 한다.
- 외부 데이터 전송 실패가 주문 처리 결과에 직접 영향을 주지 않도록 전송 책임을 분리한다.
- 인기 메뉴는 초기에는 완료 주문 데이터를 직접 집계하고, 트래픽 증가 시 별도 집계 저장소로 변경할 수 있도록 조회 책임을 분리한다.
- 동시성 제어 구현을 핵심 비즈니스 규칙과 분리하여 정합성과 성능을 비교할 수 있도록 한다.
- 확장 가능성을 이유로 현재 요구사항에 필요하지 않은 기능과 추상화를 과도하게 구현하지 않는다.
- 1차 동기 전송과 2차 Outbox + Kafka 비동기 전송을 분리하여 단계적으로 고도화한다.
- Kafka는 주문 데이터의 원본 저장소가 아니라 주문 완료 이벤트를 후속 시스템에 전달하는 메시지 브로커로 사용한다.
- Kafka 도입 후에도 주문과 포인트 정합성은 데이터베이스 트랜잭션을 기준으로 보장한다.
- Kafka 발행 실패가 완료된 주문 상태에 영향을 주지 않도록 주문 처리와 이벤트 전송을 분리한다.
- 이벤트가 중복 발행되거나 중복 소비될 수 있음을 전제로 멱등성을 보장한다.
