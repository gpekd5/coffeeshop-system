# ADR-001: 포인트 동시성 제어 전략

## 상태

제안됨

---

## 1. 배경

회원은 보유 포인트를 사용해 주문을 결제한다.

동일 회원의 주문 요청이 동시에 실행되면 두 요청이 같은 포인트 잔액을 조회한 뒤 각각 차감할 수 있다.

예를 들어 포인트 잔액이 10,000P인 회원에게 다음 주문이 동시에 들어올 수 있다.

```text
주문 A: 8,000P
주문 B: 7,000P
```

동시성 제어가 없다면 두 요청이 모두 10,000P를 기준으로 검증하여 주문 A와 주문 B가 모두 성공할 수 있다.

이 경우 실제 보유 포인트보다 많은 금액이 사용되거나, 마지막 UPDATE가 이전 결과를 덮어쓰는 Lost Update가 발생할 수 있다.

따라서 포인트 차감 시 동일 회원의 포인트 데이터에 대한 동시성 제어가 필요하다.

---

## 2. 요구사항

포인트 차감은 다음 조건을 만족해야 한다.

- 포인트 잔액은 음수가 될 수 없다.
- 동일 회원의 동시 주문에서도 초과 사용이 발생하지 않아야 한다.
- 포인트 차감과 주문 저장은 하나의 트랜잭션으로 처리되어야 한다.
- 여러 Spring Boot 인스턴스에서도 동일하게 동작해야 한다.
- 애플리케이션 인스턴스 내부의 JVM Lock에 의존하지 않아야 한다.
- 동시 주문 실패 후 주문, 결제, 포인트 이력이 일부만 저장되면 안 된다.

---

## 3. 고려한 대안

| 방식 | 동작 방식 | 장점 | 단점 | 이번 프로젝트 적합성 |
|---|---|---|---|---|
| 동시성 제어 없음 | 조회한 잔액을 기준으로 애플리케이션에서 차감 | 구현이 가장 단순하다. | Lost Update와 포인트 초과 사용이 발생할 수 있다. | 부적합 |
| 낙관적 락 | Version 값을 비교하여 동시 수정 충돌을 감지 | 락을 오래 점유하지 않는다. 충돌이 적으면 성능이 좋다. | 충돌 시 예외와 재시도 처리가 필요하다. 동일 회원 요청이 많으면 재시도 비용이 커질 수 있다. | 보통 |
| 비관적 락 | 포인트 행을 `SELECT FOR UPDATE`로 잠근 후 차감 | 같은 포인트 행의 변경을 순차 처리한다. 동작과 정합성 검증이 명확하다. | 락 대기가 발생한다. 트랜잭션이 길면 처리량이 감소하고 Deadlock 위험이 커질 수 있다. | 높음 |
| 조건부 UPDATE | `balance >= amount` 조건으로 바로 차감 | 한 번의 UPDATE로 잔액 검증과 차감을 처리할 수 있다. 락 조회보다 처리량이 좋을 가능성이 있다. | UPDATE 결과 건수를 확인해야 한다. Entity 중심 변경 방식과 다를 수 있고 트랜잭션 설계가 필요하다. | 높음 |
| Redis 분산락 | 회원 ID를 기준으로 Redis에 락을 획득한 후 DB 작업 | 여러 인스턴스가 같은 락을 공유할 수 있다. DB 이외 작업도 보호할 수 있다. | 락 만료, 해제 실패, Redis 장애를 고려해야 한다. DB 트랜잭션과 Redis 락의 생명주기를 함께 관리해야 한다. | 낮음 |
| JVM Lock | `synchronized` 또는 `ReentrantLock` 사용 | 단일 인스턴스에서는 구현이 단순하다. | 다른 애플리케이션 인스턴스와 락을 공유하지 못한다. Scale-out 환경에서 정합성을 보장하지 못한다. | 부적합 |

## 4. 결정

초기 구현에서는 **MySQL 비관적 쓰기 락**을 사용한다.

포인트 조회 시 해당 회원의 포인트 행에 쓰기 잠금을 획득한다.

```text
포인트 행 잠금 조회
→ 현재 잔액 확인
→ 잔액 검증
→ 포인트 차감
→ 주문 및 결제 저장
→ Commit
```

Repository에는 일반 조회와 잠금 조회를 구분한다.

```java
Optional<Point> findByMemberId(Long memberId);

@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("""
    select p
    from Point p
    where p.member.id = :memberId
    """)
Optional<Point> findByMemberIdForUpdate(Long memberId);
```

## 5. 결정 이유

### 5.1 포인트는 결제에 사용되는 금전성 데이터다

포인트는 단순한 부가 정보가 아니라 실제 주문 결제에 직접 사용되는 금전성 데이터다.

포인트가 한 번이라도 중복 차감되거나 실제 잔액보다 많이 사용되면 다음과 같은 문제가 발생할 수 있다.

```text
사용자의 실제 보유 포인트와 DB 잔액 불일치
주문 금액과 포인트 사용 이력 불일치
정상 사용자의 결제 실패
서비스에 대한 사용자 신뢰 하락
운영자가 수동으로 데이터를 복구해야 하는 상황 발생
```

따라서 포인트 처리에서는 높은 처리량보다 데이터 정합성을 우선해야 한다고 판단했다.

---

### 5.2 동일 회원의 포인트 행에서 충돌 가능성이 높다

포인트 차감은 동일 회원의 동일한 포인트 행을 변경한다.

예를 들어 잔액이 10,000P인 상태에서 8,000P 주문과 7,000P 주문이 동시에 실행되면 두 요청이 같은 잔액을 기준으로 검증할 수 있다.

낙관적 락은 동시 수정이 발생한 뒤 Version 충돌을 감지하고 예외와 재시도를 통해 처리한다.

하지만 포인트 차감처럼 동일 회원의 동일한 행을 대상으로 충돌이 발생할 수 있는 기능에서는 충돌 이후 재시도하는 방식보다 처음부터 행을 잠그고 요청을 순차적으로 처리하는 비관적 락이 더 명확하다고 판단했다.

```text
낙관적 락
→ 동시에 처리
→ 충돌 발생
→ 예외
→ 재시도

비관적 락
→ 먼저 요청한 트랜잭션이 행 잠금
→ 뒤의 요청은 대기
→ 앞 요청 Commit
→ 최신 잔액으로 다시 검증
```

---

### 5.3 정합성이 처리량보다 우선한다

본 프로젝트에서 포인트 초과 사용은 허용할 수 없는 데이터 오류다.

동일 회원의 요청을 일정 시간 대기시키더라도 다음 조건이 반드시 보장되어야 한다.

- 포인트 잔액이 음수가 되지 않는다.
- 보유 포인트보다 많은 금액을 사용할 수 없다.
- 주문 성공 시 포인트가 정확히 한 번 차감된다.
- 주문 실패 시 포인트 차감도 Rollback된다.
- 주문, 결제 및 포인트 이력이 서로 일치한다.

따라서 초기 구현에서는 최대 처리량보다 포인트와 주문 데이터의 정합성을 우선한다.

---

### 5.4 다중 인스턴스에서도 동일하게 동작한다

모든 Spring Boot 인스턴스가 동일한 MySQL을 사용한다.

따라서 데이터베이스 행 잠금은 요청이 어느 애플리케이션 인스턴스로 전달되더라도 동일하게 적용된다.

```text
Spring Boot Instance 1 ─┐
                        ├─ Shared MySQL Point Row
Spring Boot Instance 2 ─┘
```

애플리케이션 인스턴스 내부의 `synchronized` 또는 `ReentrantLock`은 다른 인스턴스와 잠금을 공유할 수 없다.

반면 MySQL 비관적 락은 공유 DB의 동일한 포인트 행을 기준으로 동작하므로 다중 인스턴스 환경에서도 정합성을 보장할 수 있다.

---

### 5.5 Redis 분산락이 현재 범위에는 과하다

현재 보호하려는 대상은 MySQL에 저장된 회원별 포인트 행이다.

DB 트랜잭션 내부의 포인트 변경을 보호하는 것이 목적이므로 별도의 Redis 분산락을 추가하면 다음과 같은 관리 대상이 늘어난다.

- Redis 락 획득 실패
- 락 대기 시간
- 락 만료 시간
- 트랜잭션 종료 전 락 만료
- 예외 발생 시 락 해제
- Redis 장애
- DB 트랜잭션과 Redis 락의 실행 범위 불일치

모든 인스턴스가 동일한 MySQL을 공유하고 있으며 보호 대상도 DB 행이므로, 초기 구현에서는 MySQL 비관적 락만으로 충분하다고 판단했다.

---

### 5.6 조건부 UPDATE는 비교 대상으로 유지한다

조건부 UPDATE는 잔액 검증과 차감을 하나의 SQL로 처리할 수 있다.

```sql
UPDATE points
SET balance = balance - :amount
WHERE member_id = :memberId
  AND balance >= :amount
```

UPDATE 결과가 1이면 차감 성공, 0이면 포인트 부족으로 판단할 수 있다.

비관적 락보다 조회 횟수와 락 대기 시간을 줄여 더 높은 처리량을 보일 가능성이 있다.

하지만 초기 구현에서는 동작과 정합성 검증이 명확한 비관적 락을 먼저 적용한다.

이후 동일한 부하 조건에서 비관적 락과 조건부 UPDATE를 비교하여 다음 항목을 확인한다.

- 평균 응답시간
- P95 응답시간
- 처리량
- 락 대기 시간
- 실패 요청 수
- 최종 포인트 정합성

## 6. 잠금 순서

주문 처리에서 여러 데이터를 잠글 경우 다음 순서를 유지한다.

```text
장바구니
→ 메뉴
→ 포인트
```

여러 메뉴를 잠글 경우 메뉴 ID를 오름차순으로 정렬한다.

```text
Menu 1
→ Menu 3
→ Menu 7
```

모든 주문 요청이 같은 순서로 잠금을 획득하여 Deadlock 가능성을 줄인다.

## 7. 예상 단점과 위험

### 7.1 락 대기

동일 회원의 주문이 동시에 많이 들어오면 뒤의 요청은 앞의 트랜잭션이 끝날 때까지 기다려야 한다.

### 7.2 긴 트랜잭션

트랜잭션 안에서 외부 API를 호출하거나 느린 처리를 수행하면 포인트 락이 오래 유지될 수 있다.

따라서 외부 네트워크 호출은 DB Commit 이후 실행한다.

### 7.3 Deadlock

서로 다른 요청이 장바구니, 메뉴, 포인트를 다른 순서로 잠그면 Deadlock이 발생할 수 있다.

잠금 획득 순서를 고정하고 메뉴 ID를 정렬해 조회한다.

### 7.4 전체 사용자의 병목 가능성

현재 잠금 단위는 포인트 테이블 전체가 아니라 회원별 포인트 행이다.

서로 다른 회원의 주문은 서로 다른 행을 잠그므로 병렬 처리가 가능하다.

## 8. 검증 계획

### 동일 회원 동시 주문

```text
초기 잔액: 10,000P
주문 A: 8,000P
주문 B: 7,000P
```

예상 결과:

- 주문은 한 건만 성공한다.
- 다른 주문은 포인트 부족으로 실패한다.
- 최종 잔액은 성공한 주문에 따라 2,000P 또는 3,000P다.
- 포인트 잔액은 음수가 되지 않는다.
- 성공 주문 수와 결제 및 포인트 이력 수가 일치한다.

### 여러 회원 동시 주문

예상 결과:

- 서로 다른 회원의 주문은 병렬로 처리된다.
- 한 회원의 락이 다른 회원의 포인트 변경을 막지 않는다.

### 충전과 주문 동시 실행

예상 결과:

- 실행 순서에 따라 주문 성공 여부는 달라질 수 있다.
- 충전 금액이 유실되지 않는다.
- 주문 성공 시 포인트 차감이 반드시 반영된다.
- 포인트가 음수가 되지 않는다.

### 성능 비교

다음 방식을 동일한 조건에서 비교한다.

```text
동시성 제어 없음
낙관적 락
비관적 락
조건부 UPDATE
```

측정 지표:

- 평균 응답시간
- P95 응답시간
- 처리량
- 락 대기 시간
- 실패 요청 수
- 재시도 횟수
- Deadlock 발생 수
- 최종 포인트 정합성
