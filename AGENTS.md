# AGENTS.md

## Project context

- 이 저장소는 커피 주문 및 포인트 결제 시스템입니다.
- 구현 기준은 GitHub Issue와 `docs` 폴더의 설계 문서를 우선합니다.
- Issue 범위를 벗어나는 기능은 임의로 추가하지 않습니다.
- 문서와 구현이 충돌하면 임의로 결정하지 말고 PR에 근거와 영향을 남깁니다.

## Implementation guidelines

- Entity를 API 요청 또는 응답으로 직접 사용하지 않습니다.
- Controller는 요청 검증과 응답 변환에 집중하고 비즈니스 로직을 직접 수행하지 않습니다.
- Service, Facade, Repository의 책임을 명확히 분리합니다.
- 인증 사용자는 Request Body의 `memberId`가 아니라 Security Context에서 식별합니다.
- 금액과 포인트는 서버에서 계산하며 포인트 잔액은 음수가 될 수 없습니다.
- 주문, 결제, 포인트 차감, 포인트 이력, 장바구니 삭제의 트랜잭션 범위는 `docs/ARCHITECTURE.md`와 ADR을 따릅니다.
- 외부 API 호출은 DB 트랜잭션 내부에서 실행하지 않습니다.
- 다중 서버 환경에서 동작해야 하므로 JVM 로컬 락에만 의존하지 않습니다.
- 포인트 동시성 제어는 MySQL 비관적 쓰기 락을 우선 사용합니다.
- 잠금 순서는 장바구니 → 메뉴 ID 오름차순 → 포인트 순서를 따릅니다.
- 주문 멱등성은 애플리케이션 선조회와 `(member_id, idempotency_key)` DB Unique 제약을 함께 적용합니다.
- API 응답은 `ResponseEntity<ApiResponse<T>>` 형식을 따릅니다.
- 구현 변경 시 관련 문서와 테스트도 함께 갱신합니다.

## Testing guidelines

- 각 Issue의 완료 조건과 필수 테스트를 모두 확인합니다.
- 정상 케이스뿐 아니라 실패, Rollback, 권한, 경계값 테스트를 작성합니다.
- 동시성 및 멱등성 기능은 실제 동시 요청 테스트를 작성합니다.
- 외부 연동은 성공, 실패, Timeout, 재처리 시나리오를 검증합니다.
- 작업 완료 전에 `./gradlew clean check`를 실행합니다.
- 실행하지 못한 테스트가 있다면 PR 본문에 이유와 영향을 명시합니다.

## Pull request guidelines

- PR은 하나의 Issue 범위만 다룹니다.
- PR 본문에 `Closes #이슈번호`를 포함합니다.
- 변경 내용, 테스트 결과, 설계 문서와 다른 결정, 남은 위험을 작성합니다.
- 범위를 벗어난 리팩터링이나 의존성 추가는 별도 근거가 없으면 포함하지 않습니다.

## Review guidelines

- 모든 리뷰 요약과 인라인 코멘트는 한국어로 작성합니다.
- 코드 식별자, 클래스명, 메서드명, 예외 메시지와 로그 원문은 원래 표기를 유지합니다.
- GitHub Issue의 구현 범위와 제외 범위를 먼저 확인합니다.
- `docs/API_SPEC.md`, `docs/BUSINESS_RULES.md`, `docs/ERD.md`, `docs/ARCHITECTURE.md` 및 ADR과 구현이 일치하는지 확인합니다.
- Controller, Service, Facade, Repository의 책임이 적절히 분리되어 있는지 확인합니다.
- Entity가 API DTO로 직접 노출되지 않는지 확인합니다.
- 트랜잭션 범위와 Rollback 가능성을 확인합니다.
- 외부 API 호출이 DB 트랜잭션 내부에서 실행되지 않는지 확인합니다.
- 포인트 잔액이 음수가 되거나 중복 차감될 가능성이 없는지 확인합니다.
- 주문 멱등성과 DB Unique 제약이 올바르게 적용되는지 확인합니다.
- 인증 사용자 식별, 관리자 권한, 토큰 및 개인정보 로그 노출 여부를 확인합니다.
- HTTP 상태 코드, 오류 코드, 공통 응답 형식이 API 명세와 일치하는지 확인합니다.
- 정상 케이스뿐 아니라 실패, Rollback, 동시성, 멱등성 테스트가 작성되었는지 확인합니다.
- 실제 장애, 데이터 손상, 보안 문제로 이어질 수 있는 항목을 우선 지적합니다.
- 단순 취향이나 사소한 스타일 차이는 리뷰하지 않습니다.
- 문제를 지적할 때 원인, 영향, 발생 조건과 가능한 수정 방향을 한국어로 설명합니다.
- 문제가 없다면 불필요한 지적을 만들지 말고 한국어로 간단히 승인 의견을 작성합니다.
